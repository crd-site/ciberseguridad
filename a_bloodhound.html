☠️ Laboratorio: BloodHound — Reconocimiento Ofensivo de Active Directory y Rutas de Escalada (Red & Blue Team)

Este laboratorio demuestra cómo instalar y usar BloodHound Community Edition (CE) para mapear la infraestructura de identidad de un dominio Active Directory, identificar rutas de ataque hacia Domain Admin y comprender las relaciones de privilegios entre objetos del dominio usando el entorno GOAD (Game of Active Directory).

🛡️ Mapeo de Tácticas y Técnicas (MITRE ATT&CK)

A continuación se detallan las técnicas adversariales que se implementarán en este ejercicio:

Táctica Técnica Código MITRE
Reconocimiento Gather Victim Identity Information: Credentials T1589.001
Descubrimiento Domain Trust Discovery T1482
Descubrimiento Permission Groups Discovery: Domain Groups T1069.002
Descubrimiento Account Discovery: Domain Account T1087.002
Movimiento Lateral Use Alternate Authentication Material T1550

🏗️ Escenario y Arquitectura de Red

Para que el laboratorio sea exitoso, es fundamental configurar el entorno virtual de la siguiente manera:

  • Segmento de Red: Todas las VMs (dc01, dc02, srv02 y Kali) deben estar conectadas al switch virtual VMnet5 (Host-only).
  • Subred: El laboratorio opera en el rango 192.168.56.0/24 o de acuerdo como tienes configurado el Virtual Network Editor
  • Orden de arranque: Encender siempre en este orden: dc01 → dc02 → srv02 → Kali.
  • Verificación de conectividad:
    • Desde Kali, ejecutar ping 192.168.56.10 para verificar conectividad con dc01.
    • Desde Kali, ejecutar ping 192.168.56.11 para verificar conectividad con dc02.
  • RAM recomendada para Kali: Mínimo 8 GB asignados en VMware Settings para que BloodHound CE funcione correctamente.
VM Nombre Dominio IP
dc01 KINGSLANDING sevenkingdoms.local 192.168.56.10
dc02 WINTERFELL north.sevenkingdoms.local 192.168.56.11
srv02 CASTELBLACK north.sevenkingdoms.local 192.168.56.22
Kali Atacante — 192.168.56.x (DHCP)

🔧 FASE 1: Instalación de BloodHound CE en Kali Linux

1. Instalar Docker

sudo apt update
sudo apt install -y docker.io docker-compose
sudo systemctl start docker
sudo systemctl enable docker
sudo usermod -aG docker $USER
newgrp docker
  • Explicación: Instala Docker y Docker Compose, habilita el servicio y agrega el usuario al grupo docker para no requerir sudo en cada comando.
  • Verificación: Ejecutar docker --version — debe mostrar Docker version 24.x o superior.

2. Descargar BloodHound CE

mkdir -p ~/Desktop/laboratorio
cd ~/Desktop/laboratorio
git clone https://github.com/SpecterOps/BloodHound.git bloodhound
cd bloodhound
  • Explicación: Clona el repositorio oficial de BloodHound CE desde GitHub en la carpeta de trabajo del laboratorio.

3. Configurar BloodHound CE

cp examples/docker-compose/docker-compose.yml docker-compose.yml
  • Explicación: Copia el archivo de configuración de ejemplo. Este archivo define los tres contenedores que necesita BloodHound: la aplicación, Neo4j y PostgreSQL.

4. Iniciar BloodHound CE

docker compose up -d
  • Explicación: Inicia los tres contenedores en segundo plano. La primera vez descarga las imágenes Docker (~2 GB) — puede tardar entre 5 y 10 minutos.
  • Verificación: Ejecutar docker ps — deben aparecer 3 contenedores en estado healthy.

5. Obtener la contraseña inicial

sleep 30
docker logs bloodhound-bloodhound-1 2>&1 | grep -i "initial"
  • Explicación: BloodHound CE genera una contraseña aleatoria en el primer arranque. Esta contraseña solo aparece una vez en los logs — anotarla antes de continuar.
  • Salida esperada: # Initial Password Set To: a9TnkXHJyevmiGYOUSbC316VOFzjqrIx
⚠️ Importante: Si el contenedor de BloodHound cae al hacer login, la causa más probable es falta de RAM. Apagar todas las VMs de GOAD y reiniciar Kali con al menos 8 GB asignados en VMware Settings antes de continuar.

Si perdiste la contraseña inicial, recrea los contenedores con: docker compose down -v && docker compose up -d y vuelve a leer los logs.

6. Primer login en BloodHound CE

Abrir el navegador en Kali y navegar a:

http://localhost:8080
  • Email: admin
  • Password: la contraseña obtenida en el paso anterior
  • BloodHound pedirá cambiar la contraseña en el primer login — elegir una que sea fácil de recordar para el laboratorio.

7. Instalar el colector bloodhound-python

pip3 install bloodhound-ce --break-system-packages
  • Explicación: Instala la versión del colector compatible con BloodHound CE v9. La versión legacy instalada con apt install bloodhound genera archivos incompatibles con CE v9 y no debe usarse.
  • Verificación: Ejecutar bloodhound-python --help — debe mostrar BloodHound.py for BloodHound Community Edition.

🎯 FASE 2: Recolección de Datos del Dominio

8. Configurar DNS en Kali

Para que bloodhound-python pueda resolver los nombres de host del dominio, configurar el DNS de Kali apuntando al Domain Controller:

DNS 1: 192.168.56.10

DNS 1: 192.168.56.11

Probar la resolucion de nombres con:

nslookup winterfell.north.sevenkingdoms.local
# Debe mostrar: Address: 192.168.56.11

9. Recolectar datos del dominio norte

cd ~/Desktop/laboratorio
bloodhound-python \
  -u jon.snow \
  -p iknownothing \
  -d north.sevenkingdoms.local \
  -ns 192.168.56.11 \
  -c All \
  --zip
  • -u / -p: Credenciales de un usuario estándar del dominio — no se necesitan privilegios de administrador.
  • -d: Nombre del dominio a recolectar.
  • -ns: Servidor DNS — la IP del Domain Controller.
  • -c All: Recolecta todos los tipos de datos: usuarios, grupos, equipos, GPOs, ACLs, sesiones y trusts.
  • --zip: Empaqueta todos los archivos JSON en un único .zip para facilitar la importación.
  • Salida esperada: INFO: Done in 00M 03S — INFO: Compressing output into 20260422150106_bloodhound.zip

10. Recolectar datos del dominio raíz (opcional)

bloodhound-python \
  -u jon.snow \
  -p iknownothing \
  -d sevenkingdoms.local \
  -ns 192.168.56.10 \
  -c All \
  --zip
  • Explicación: Recolecta datos del dominio padre. Al importar ambos ZIPs en BloodHound, el grafo mostrará rutas de ataque que cruzan la relación de confianza entre dominios.

🔍 FASE 3: Análisis en BloodHound CE

11. Importar datos en BloodHound CE

  1. Abrir http://localhost:8080 e iniciar sesión.
  2. Hacer clic en Quick Upload en el menú lateral izquierdo.
  3. Arrastrar el archivo .zip generado o hacer clic para seleccionarlo.
  4. Esperar que BloodHound procese los datos — verás la barra de progreso al 100%.
  5. Cuando muestre "All files have successfully been uploaded for ingest", hacer clic en Close.
  6. Hacer clic en Explore en el menú lateral.

12. Buscar Domain Admins y analizar el grafo

En el buscador Search Nodes escribir Domain Admins y seleccionar el resultado. En el panel derecho expandir:

  • Members: cuántas cuentas son Domain Admin.
  • Inbound Object Control: cuántos objetos tienen control SOBRE Domain Admins — cada uno es una ruta de ataque potencial.
  • Outbound Object Control: cuántos objetos controla Domain Admins — este número representa el radio de explosión si un atacante lo compromete.

13. Calcular ruta de ataque con Pathfinding

Hacer clic en la pestaña PATHFINDING en la parte superior y configurar:

Start Node: JON.SNOW@NORTH.SEVENKINGDOMS.LOCAL
End Node:   DOMAIN ADMINS@NORTH.SEVENKINGDOMS.LOCAL
  • Resultado esperado: BloodHound mostrará una ruta de 4 saltos: jon.snow → AllowedToDelegate → WINTERFELL → CoerceToTGT → north.sevenkingdoms.local → Contains → USERS → Contains → DOMAIN ADMINS
  • Conclusión pedagógica: Un usuario estándar sin privilegios aparentes tiene una ruta directa hacia Domain Admin explotando una delegación Kerberos mal configurada.

14. Consulta Cypher — encontrar cuentas Kerberoasteables

Hacer clic en la pestaña CYPHER y ejecutar:

MATCH (u:User) WHERE u.hasspn = true
RETURN u.name, u.admincount
ORDER BY u.admincount DESC
  • Resultado esperado: Lista de cuentas con SPNs registrados — en GOAD aparecerán sansa.stark y sql_svc, ambas vulnerables a Kerberoasting.

📊 Resumen de Técnicas MITRE ATT&CK

  • T1589.001 - Credentials: Uso de credenciales de usuario estándar para recolección masiva de datos del dominio.
  • T1482 - Domain Trust Discovery: BloodHound mapea la relación de confianza padre-hijo entre sevenkingdoms.local y north.sevenkingdoms.local.
  • T1069.002 - Domain Groups: Enumeración de grupos privilegiados como Domain Admins y sus miembros.
  • T1087.002 - Domain Account: Descubrimiento de todos los usuarios del dominio y sus atributos de seguridad.
  • T1550 - Alternate Authentication: Identificación de rutas de ataque mediante delegación Kerberos (AllowedToDelegate).

2. Recolección de datos — bloodhound-python

Parámetro Descripción Ejemplo
-u / -p Usuario y contraseña del dominio. -u jon.snow -p iknownothing
-d Nombre del dominio a recolectar. -d north.sevenkingdoms.local
-ns Servidor de nombres — IP del Domain Controller. -ns 192.168.56.11

3. Navegación en la interfaz BloodHound CE

Panel / Función Descripción Uso
Search Busca nodos por nombre en el grafo. Escribir Domain Admins o user:jon.snow
Pathfinding Calcula la ruta más corta entre dos nodos. Start: usuario comprometido — End: Domain Admins
Cypher Ejecuta consultas directas sobre la base de datos de grafos Neo4j. Consultas personalizadas para encontrar vulnerabilidades específicas

💡 Lecciones Aprendidas

  • Una cuenta estándar es suficiente: bloodhound-python solo necesita una cuenta de usuario normal para mapear completamente toda la infraestructura de identidad del dominio — no requiere privilegios de administrador.
  • Los permisos acumulados son el problema real: Las configuraciones inseguras no son visibles revisando permisos uno por uno — BloodHound los encadena y muestra la ruta completa automáticamente.
  • La dirección de las flechas importa: En el grafo, la dirección de cada arista define quién controla a quién. Confundirla lleva a conclusiones incorrectas sobre las rutas de ataque.
  • El radio de explosión es el indicador defensivo clave: El Outbound Object Control de Domain Admins equivale a todos los objetos del dominio — en producción pueden ser miles, lo que explica por qué comprometer DA es catastrófico.
  • Los datos se acumulan: Cada Quick Upload agrega información al grafo sin borrar la anterior — importar ambos dominios juntos revela rutas cross-domain que no serían visibles con un solo dominio.
